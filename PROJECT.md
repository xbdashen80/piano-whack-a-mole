# 钢琴打地鼠 · 项目开工文档

> 这是一份给 Claude Code 的接手说明。请先完整读一遍，再开始动手。
> 现有的可运行原型是单文件 `钢琴打地鼠_小熊滑板版.html`（与本文件放在同一目录）。
> 第一阶段的核心任务是：把它拆成可维护的模块结构，并在能实际运行的环境里彻底解决“卡死”这个顽固 bug。

---

## 1. 这是什么

一个把**电钢琴变成游戏手柄**的网页游戏。面向**零基础、完全不懂钢琴**的用户（我本人）。
核心理念：玩着玩着，把琴键位置、手指指法、节奏感顺便练出来——寓教于乐，游戏性优先。

玩法是“打地鼠 + 命脉小熊”：
- 屏幕下方画出一排琴键，和真琴上的键**位置一一对应**。
- 键的上方随机冒出**目标圈**，圈外有一道**倒计时弧线**在缩。
- 玩家要在弧线缩没之前，用圈里标注的**手指编号**，按下真琴上对应的那个键。
- 按对 → 加分、连击、爆粒子特效、悦耳音效。
- 屏幕左侧有一只**卡通小熊踩滑板浮在水面上**：每按对一下滑板就把它往上弹一截；不按它就缓缓下沉；越靠近水面小熊表情越害怕（开心→紧张→惊恐）；沉进水里 = 失败（相当于“命/血条”的拟人化）。
- 顶部有**过关进度条**：本关得分达到目标分即**通关**，解锁下一关。

## 2. 用户与硬件背景（很重要，影响设计取舍）

- 用户是**钢琴零基础**，所以引导、指法提示、键位提示要足够直白；难度起点要非常低。
- 钢琴是 **Yamaha P-115**，背后有 `USB TO HOST` 方口，通过 USB 打印线接电脑，走 **Web MIDI**，识别精确、零延迟。
- 浏览器用 **Chrome / Edge**（Web MIDI 仅这些支持）。
- 没插琴时，用电脑键盘 `A S D F G H J K L` 模拟琴键，方便开发调试。

## 3. 当前技术栈

- 纯前端，单个 HTML 文件，无构建工具。
- **Web MIDI API**：读取琴键按下事件（note on）。
- **Tone.js**（CDN 引入）：合成背景音乐（鼓机 + 贝斯 + 旋律）和命中音效。
- **Canvas 2D**：画键盘、目标圈、小熊动画、粒子特效、水面。
- **localStorage**：存档（最高分、已解锁关卡）。
- 游戏主循环用 `requestAnimationFrame`。

## 4. 现有功能清单（原型已实现）

- Web MIDI 连接 + 键盘兜底输入。
- 打地鼠核心循环：随机生成目标、倒计时判定、按对/漏掉/按错的处理。
- 琴键渲染：音名（C D E…）+ 手指编号（1拇指 2食指 3中指 4无名指 5小指，带配色）。
- 命脉小熊：踩滑板浮水面、按对上弹、随时间下沉、表情随危险度变化、掉水里失败动画。
- 30 个关卡：按键数（3→12）和速度细分，梯度平缓。每关有过关目标分。
- 5 档递进背景音乐（欢快/放克/舞曲/快板/终章，118→150 BPM），随关卡自动切换。
- 连击系统、得分、过关进度条、通关/失败结算页、关卡选择网格、开始界面键位预览。
- 背景音乐、音效的独立开关。

## 5. 已知问题（按优先级）

### P0 —— 游戏会“卡死/没反应”（最高优先，必须先解决）
现象：玩到某一关中途（用户报告过第 2 关、第 3 关），地鼠不再出现、按键无响应，游戏像是僵住。
单文件版里反复出现，**靠读代码盲改修不干净**，这正是需要在能实际运行、能看控制台的环境里定位的根本原因。
已尝试但不彻底的措施：给关键函数加 try/catch；给生成逻辑加“追赶式补足 + 看门狗强制生成”。
**接手后请：在本地真实运行，复现卡死，用控制台/断点找到真正的根因**（怀疑方向：requestAnimationFrame 与 Tone.js 调度/AudioContext 挂起的交互、标签页失焦后计时器漂移、状态机在过关切关时的竞态、moles 数组在某状态下不再被消费）。**先复现，再修，最后加一个自动化的冒烟测试防止回归。**

### P1 —— 背景音乐的递进做法
之前一版用“放慢节奏”表现简单关卡，导致太慢没法玩。现版改为：各档都保持较快 BPM，靠**风格/鼓型/旋律**不同来体现递进，而不是靠慢。希望进一步打磨：每档要更像“一首能循环听的小段落”，可考虑引入更真实的鼓/合成器采样音色。

### P2 —— 想做但还没做的功能
- **节奏同步**：让目标圈踩着背景鼓点的拍子冒出来，玩家按中正好卡在节拍上 → 从“打地鼠”升级为真正的“音乐节奏游戏”。这是用户最期待的下一步。
- 更真实的音色（鼓采样、钢琴采样）让声音更“饱满、有游戏机感”。
- 可能的扩展：曲库 / 跟弹真实旋律模式 / 关卡编辑器 / 每日练习统计。

## 6. 第一阶段任务（建议顺序）

1. **用 git 初始化项目**，把现有 HTML 作为初始提交存档。
2. **拆分模块**（仍可保持零构建、用 ES modules，或按需引入极简打包工具，由你判断）。建议拆成：
   - `midi.js`：Web MIDI 接入 + 键盘兜底，对外发出统一的“按键事件”。
   - `audio.js`：Tone.js 封装，背景音乐档位、音效。
   - `levels.js`：关卡数据 + 音乐档位映射（纯数据，方便调参）。
   - `bear.js`：小熊 + 滑板 + 水面的绘制与物理。
   - `game.js`：主循环、状态机、判定、计分、过关/失败流程。
   - `ui.js`：HUD、进度条、结算页、关卡选择、开始界面。
   - `main.js` / `index.html`：装配。
3. **复现并彻底修掉 P0 卡死 bug**，加一个最小冒烟测试（例如用模拟时钟跑若干分钟主循环，断言地鼠持续生成、状态不僵死）。
4. 跑通后再继续 P1 / P2。

## 7. 设计原则（请始终遵守）

- **零基础友好**：起点极低，提示直白，绝不假设用户懂乐理。
- **游戏性第一**：即时反馈、紧张感、明确的过关目标和成长解锁。
- **声音要饱满**：背景律动 + 清脆有冲击的命中音效，要有“游戏机/街机”的存在感，不能干、不能空。
- **梯度平缓**：相邻关卡只难一点点，不能出现“突然跟不上”。
- **改动增量化、可回退**：每个改动小步提交，方便 debug 和回滚。

## 8. 验收标准（第一阶段）

- [ ] 项目已模块化拆分，结构清晰，git 历史可用。
- [ ] 连续游玩、反复过关切关 ≥ 10 分钟不再出现“卡死/没反应”。
- [ ] 有一个能跑的冒烟测试覆盖主循环不僵死。
- [ ] 在 Chrome 下插上 Yamaha P-115 能正常连 MIDI、正常游玩；未插琴时键盘兜底可用。
- [ ] 五档背景音乐切换正常，第一关音乐即已饱满可玩。

### 附：本地运行说明
- 用 Chrome/Edge 打开 `index.html`（或拆分后的入口）。
- Web MIDI 与 AudioContext 需要用户手势触发，首次点“开始”时初始化音频。
- 若用 ES modules，需通过本地静态服务器打开（如 `python -m http.server`），不能用 `file://` 直接打开。
